# Mail 桌面端程序方案（Rust + Tauri）

## 1. 项目目标

为现有 Mail 邮箱管理系统提供 Windows 桌面客户端，最终体验接近 QQ 邮箱桌面版：

- 用户双击桌面图标启动独立窗口，不打开系统浏览器。
- 用户使用现有账号登录，读取服务器数据库中属于自己的邮箱账号、邮件、公告和设置。
- 数据库、邮箱凭据和服务器密钥始终保留在服务端，客户端不直接连接 SQLite。
- 支持登录状态持久化、系统托盘、桌面通知、单实例运行和自动更新。
- 首期支持 Windows 10/11 x64，后续可扩展到 Windows ARM64、macOS 和 Linux。

桌面端采用 **Rust + Tauri 2 + WebView2**。现有 React/TypeScript 前端继续复用，当前 Go 后端继续作为唯一业务服务和数据库访问层。

## 2. 为什么选择 Rust + Tauri

相较 Electron，Tauri 更适合本项目的原因：

- Windows 使用系统 WebView2，不需要在安装包中携带完整 Chromium。
- 安装包和运行内存通常明显小于 Electron。
- Rust 主进程适合实现窗口、托盘、通知、更新和安全存储。
- 可以继续使用现有 React 页面，避免重新开发完整邮箱界面。
- Tauri 权限能力可以按白名单开放，减少远程页面获得本机权限的风险。
- 后续需要更强桌面能力时，可以逐步将网络请求、凭据保存和后台任务迁移到 Rust。

需要注意：Tauri 仍然使用 WebView 展示界面，并不是使用 Rust 重写全部 UI。Rust 主要负责桌面运行时和原生能力。

## 4. 分阶段实现策略

### 4.1 第一阶段：远程安全壳 MVP

Tauri 主窗口直接加载：

```text
https://www.aillive.xyz/mail/
```

该模式最适合快速生成第一个可用 EXE：

- 现有页面和 Go API 基本不需要修改。
- 页面和 API 同源，现有 HttpOnly Cookie 登录机制可以继续使用。
- WebView2 的用户数据目录保存 Cookie，用户再次启动时可以保持登录。
- 网站发布新版本后，桌面端界面自动获得更新，无需每次重新打包 EXE。
- 远程页面默认不获得 Tauri IPC 和本机命令权限。

第一阶段限制：

- 断网时无法进入主界面。
- 页面版本完全跟随线上站点。
- 远程页面不能直接调用高权限 Tauri 插件，否则会扩大安全边界。
- 桌面通知和托盘未读数需要通过 Rust 后台轮询或后端推送单独实现。

### 4.2 第二阶段：本地前端正式版

将现有 React 前端构建后打包进 Tauri，页面从客户端本地资源加载，业务数据仍从远程 Go API 获取。

推荐数据通路：

```text
本地 React 页面
      │ Tauri invoke
      ▼
Rust API Client（reqwest）
      │ HTTPS
      ▼
https://www.aillive.xyz/mail/api
```

这样做的优点：

- 前端资源随客户端版本固定，能够进行灰度发布和版本回滚。
- 网络请求集中由 Rust 管理，不需要给 WebView 配置宽泛 CORS。
- Refresh Token 或桌面会话可保存在 Windows Credential Manager。
- 可以实现离线启动页、网络错误页和本地只读缓存。
- 只有本地可信前端可以调用经过白名单限制的 Tauri 命令。

第二阶段需要后端增加桌面认证接口，建议采用短期 Access Token 和可轮换 Refresh Token，而不是将现有 Cookie 直接复制到本地文件。

## 7. 第一阶段窗口设计

主窗口建议参数：

- 默认尺寸：`1380 × 880`
- 最小尺寸：`1000 × 680`
- 默认居中
- 启动时隐藏菜单栏
- 窗口标题：`Aillive Mail`
- 使用项目现有纸飞机 Logo 生成 Windows ICO
- 关闭按钮默认最小化到托盘，托盘菜单提供“打开”和“退出”
- 再次启动 EXE 时激活现有窗口，不创建第二个进程

Rust 入口逻辑示意：

```rust
const APP_URL: &str = "https://www.aillive.xyz/mail/";
const ALLOWED_HOST: &str = "www.aillive.xyz";

// 1. 注册单实例插件。
// 2. 创建 WebViewWindow，并加载 APP_URL。
// 3. 只允许在应用窗口内访问 HTTPS + ALLOWED_HOST。
// 4. Microsoft 授权页和其他外部网址交给系统默认浏览器。
// 5. 不为远程页面开放任意文件、Shell 或系统命令权限。
```

具体 API 以项目安装的 Tauri 2.x 版本为准，在初始化时使用 `WebviewUrl::External` 创建窗口。

### 7.1 登录页面视觉规范（强制）

桌面端未登录时必须显示独立、完整的 Mail 登录页面，视觉结构以需求参考图为准，不允许直接显示普通网页浏览器框架，也不允许在登录页提前露出邮箱列表、侧边栏或主界面内容。

登录页面从上到下包含：

1. 居中的纸飞机 Logo 与 `Mail` 品牌文字。
2. “登录 / 注册”双标签切换区。
3. 邮箱输入框。
4. 密码输入框。
5. 黑色主登录按钮，按钮文字为“登录 Mail”。
6. “或者”分隔线。
7. “以游客模式继续”按钮。
8. 右上角可保留语言与主题切换，但不能干扰主登录区域。

桌面宽屏参考尺寸：

```text
登录内容最大宽度：约 690px
品牌区域高度：约 110px
标签栏高度：约 58px
输入框高度：约 72px 至 78px
主按钮高度：约 72px 至 80px
圆角：约 12px 至 14px
```

当窗口高度不足时允许使用紧凑布局，但必须保持相同结构、对齐关系和品牌比例，不能出现纵向内容被裁切。

视觉要求：

- 页面主体为简洁浅色背景，深色模式使用对应深色变量。
- 输入框使用轻微蓝灰色背景，与页面背景形成柔和层级。
- 品牌飞机必须使用项目统一的 `paper-plane-logo.png`，不能使用另一套不一致图标。
- 表单错误在对应区域显示，登录失败时不能播放成功转场。
- 登录请求处理中锁定重复提交，并在按钮内显示明确加载状态。
- 密码字段始终使用密码类型，不显示明文。

现有 React 实现中的 `LoginPage`、`.login-page`、`.login-card`、`.login-brand-hero`、`.auth-tabs` 和 `.login-submit` 作为改造基础，不重新建立第二套登录页面。

### 7.2 登录页飞机背景规范（强制）

登录页面可以使用低透明度的纸飞机飞行动画作为背景氛围，但背景飞机必须满足：

- 只存在于未登录页面。
- 飞机图片和动画容器必须使用透明背景，不允许出现白色方块、彩色底板或矩形图片边界。
- “无底色”指飞机素材及飞行动画图层透明；登录窗口本身仍使用当前主题的页面背景色。
- 位于登录表单和品牌 Logo 下方，不遮挡输入框、按钮和文字。
- 数量受控，建议同时可见不超过 3 至 5 架。
- 使用低透明度、慢速移动和轻微缩放，不制造视觉噪音。
- 不响应鼠标事件，不参与键盘焦点。
- 登录失败时继续保持背景状态，不错误消失。
- 登录成功后立即进入统一退出流程，所有背景飞机必须完全移除。

背景飞机应放在独立容器中，例如：

```text
.login-flight-background
```

该容器必须保持 `background: transparent`，内部飞机继续使用带 Alpha 通道的 `paper-plane-logo.png`。当前 `public/paper-plane-logo.png` 已是 `32bpp ARGB` 透明素材，可以直接复用，不需要重新制作带白底版本。

成功登录后该容器先停止生成新飞机，再在约 `180ms` 至 `300ms` 内降低透明度并卸载 DOM。主品牌飞机不属于该背景容器，必须单独保留用于飞入主界面。

### 7.3 登录成功飞机转场（强制）

登录或游客模式成功后，必须在同一个 Tauri 主窗口内完成以下连续动画，不创建新窗口，不打开浏览器：

飞机过渡动画必须保持当前 React 实现一致，包括现有飞行路径、缩放节奏、转向方式、落点计算和主界面揭示方式。本阶段不重新设计飞行动作，只处理动画图层透明背景和 Tauri 桌面环境适配。

```text
登录成功
  → 锁定表单交互
  → 所有背景飞机淡出并卸载
  → 登录卡片和 Mail 文字淡出
  → 保留品牌区域中的主纸飞机
  → 主纸飞机沿曲线路径飞行
  → 主界面从飞机最终落点开始揭示
  → 飞机落到桌面主界面左上角品牌位置
  → 切换为主界面真实 Logo
  → 加载用户邮箱数据
```

桌面端最终落点固定为主界面左侧栏顶部 `.brand-mark img`。在当前布局中目标尺寸约为 `31 × 31px`，动画必须根据实际窗口尺寸和 DOM 坐标动态计算，不能写死单一屏幕坐标。

动画要求：

- 主飞机从登录品牌 Logo 的真实屏幕位置起飞。
- 使用三次贝塞尔曲线或等价平滑路径，不使用直线瞬移。
- 完整转场总时长按当前体验固定为约 `5.2s`，计时范围从服务端确认登录成功开始，到桌面主界面完成揭示并恢复交互结束。
- 其中约 `2.8s` 只是纸飞机沿路径飞行的分段时长，后续还包含约 `420ms` 的落点停留，以及主界面揭示和稳定阶段；不得把 `2.8s` 误写为完整转场时长。
- 保持当前飞行过程中的放大、转向和阴影变化，不调整现有节奏。
- `.login-plane-motion`、`.login-plane-body` 和飞机图片本身必须为透明背景。
- 动画中只渲染纸飞机可见像素和必要投影，不允许出现白底、圆形色块、卡片底板或矩形遮罩。
- 当前 `.login-page.auth-transitioning::after` 的径向底色/光晕不作为飞机底板使用；桌面版应移除该有色遮罩或保持完全透明。
- 登录卡片在飞机起飞后淡出，不能与主界面长时间重叠。
- 主界面应从飞机落点附近向外揭示，避免整屏突然闪现。
- 飞行中的临时飞机和主界面真实 Logo 不能同时可见造成“双飞机”。
- 动画完成后必须清理定时器、临时 DOM、CSS 变量和过渡状态。
- 动画期间禁止再次点击登录、注册、游客按钮或切换页面。
- 如果无法取得飞机或目标元素坐标，应安全跳过动画并直接进入主界面，不能卡死登录。

当前代码中的以下能力应继续复用并进行桌面端验证：

- `beginSuccessTransition`：计算起点、终点和飞行路径。
- `brandPlaneRef`：登录品牌飞机起点。
- `desktopFlightTargetRef`：桌面侧边栏 Logo 落点。
- `login-plane-motion`：飞行容器。
- `login-plane-distance` / `login-plane-body`：路径、缩放与旋转动画。
- `appEntering` / `appRevealReady`：主界面接管和揭示状态。

### 7.4 转场状态机

为避免网络请求、动画和页面渲染互相竞争，登录流程必须按状态机实现：

```text
idle
  └─ 用户提交 → authenticating

authenticating
  ├─ 登录失败 → error → idle
  └─ 登录成功 → background-exiting

background-exiting
  └─ 背景飞机移除 → plane-flying

plane-flying
  └─ 主飞机到达目标 → app-revealing

app-revealing
  └─ 主界面完成揭示 → ready
```

正常动画模式下，从 `background-exiting` 开始到 `ready` 的总时间应控制在 `5.2s ± 0.2s`。验收测试以完整状态机总时长为准，而不是只读取 CSS 中单个 `animation-duration`。

只有进入 `ready` 后才恢复完整交互。邮箱账号和邮件数据可以预取，但不得因为接口较慢破坏飞机转场；主界面可在落地后显示局部加载状态。

如果系统启用了“减少动态效果”，仍需保留“登录页消失 → Logo 交接 → 主界面出现”的语义，但可缩短为约 `200ms` 至 `350ms` 的淡入淡出，不执行长距离飞行。

## 8. 导航与域名规则

允许在客户端窗口内加载：

```text
https://www.aillive.xyz/mail/
https://www.aillive.xyz/mail/*
```

以下链接应由系统默认浏览器打开：

- Microsoft OAuth 和 Device Code 授权页面
- 隐私政策、帮助中心等站外页面
- 用户点击的邮件正文外部链接

以下导航应直接阻止：

- `http://` 非加密地址
- `file://`
- `javascript:`
- 未在白名单中的自定义协议
- 非 `www.aillive.xyz` 域名试图替换主窗口页面

不要因为证书错误而忽略 TLS 校验。证书异常时应显示连接错误页并停止加载。

## 9. 登录与会话方案

### 9.1 第一阶段

继续使用当前网页端 HttpOnly Cookie：

- 登录请求由线上页面提交到同源 `/mail/api/auth/login`。
- Cookie 保存到 Tauri/WebView2 独立用户数据目录。
- Cookie 不暴露给 Rust 或前端 JavaScript。
- 退出登录继续调用现有 `/mail/api/auth/logout`。
- 卸载程序默认保留用户数据；可提供“清除登录数据”功能。

每个 Windows 用户应使用独立的数据目录，例如：

```text
%LOCALAPPDATA%\AilliveMail\WebView2
```

### 9.2 第二阶段

建议新增接口：

```text
POST /api/desktop/auth/login
POST /api/desktop/auth/refresh
POST /api/desktop/auth/logout
GET  /api/desktop/auth/status
```

推荐规则：

- Access Token 有效期约 10 至 20 分钟，只保存在内存。
- Refresh Token 随使用轮换，只保存在 Windows Credential Manager。
- 服务端数据库保存 Refresh Token 哈希、设备 ID、到期时间和撤销状态。
- 用户可以在网页端查看并注销已登录的桌面设备。
- Token 必须绑定用户和设备，不包含邮箱密码或 Microsoft Refresh Token。
- Rust 发现 `401` 时只自动刷新一次，失败后返回登录页。

## 10. Rust 与前端通信

第一阶段远程页面不需要调用 Tauri IPC，这是最安全的默认状态。

第二阶段仅开放必要命令，例如：

```text
auth_login
auth_logout
auth_status
api_request
open_external
show_notification
get_app_version
check_for_updates
```

禁止开放：

- 任意 Shell 命令执行
- 任意文件路径读取或写入
- 任意 URL 网络代理
- 返回系统凭据原文
- 允许前端指定可执行文件路径

`api_request` 必须验证路径，只允许访问预定义 API 前缀，不能接受完整任意 URL。

## 11. 前端适配方案

当前前端的请求集中在 `src/api.ts`，适合保留网页和桌面双模式。

建议后续抽象为：

```text
api(path, init)
├─ Web 模式：fetch(相对路径)
└─ Tauri 模式：invoke("api_request", ...)
```

通过 Vite 环境变量区分：

```text
VITE_RUNTIME=web
VITE_RUNTIME=tauri
```

第一阶段不需要修改该请求层；第二阶段才启用 Tauri 分支。

## 12. 系统托盘与后台行为

托盘菜单建议包含：

- 打开 Mail
- 同步邮件
- 显示未读数量
- 检查更新
- 退出

窗口行为：

- 用户点击关闭时默认隐藏到托盘。
- 用户选择“退出”才真正结束进程。
- 托盘双击恢复窗口。
- 新邮件通知点击后打开对应账号和邮件。

第一阶段可以先只实现打开和退出。未读同步需要后端提供轻量状态接口，或在 Rust 中维护受控轮询。

## 13. 桌面通知

建议新增轻量接口：

```text
GET /api/desktop/sync-status?since=<cursor>
```

返回内容只包含：

- 账号 ID
- 新邮件数量
- 发件人显示名
- 邮件主题摘要
- 下一次同步游标

不要将完整邮件正文写入桌面日志或通知缓存。通知内容应允许用户在设置中选择“显示摘要”或“仅提示有新邮件”。

## 14. 自动更新

采用 Tauri 官方 updater：

- 更新包必须使用离线私钥签名。
- 更新公钥可打包进客户端。
- 私钥不得提交到 Git 仓库或放入服务器公开目录。
- 更新元数据可部署到 `https://www.aillive.xyz/mail/download/latest.json`。
- 支持稳定版和测试版两个更新通道。
- 更新失败不能影响当前版本继续启动。

版本号统一使用语义化版本，例如：

```text
1.0.0
1.1.0-beta.1
```

## 15. Windows 安装与发布

首期发布目标：

- `Aillive Mail Setup.exe`：NSIS 安装包，推荐用户使用。
- `Aillive Mail.msi`：企业或批量部署版本，可在后续提供。
- x64 为首个正式架构。

正式发布应配置 Windows 代码签名证书，否则 SmartScreen 可能显示“未知发布者”。

构建环境要求：

- Windows 10/11
- Visual Studio Build Tools（C++ MSVC 工具链）
- Rust stable-x86_64-pc-windows-msvc
- Node.js 24+
- npm 11+
- WebView2 Runtime

预期构建命令：

```powershell
npm run tauri:dev

```

最终文件由 Tauri 输出到类似目录：

```text

src-tauri\target\release\bundle\msi\
```

## 16. 配置建议

建议在前端和 Tauri 配置中统一定义：

```text
产品名称：Aillive Mail
应用标识：xyz.aillive.mail
线上入口：https://www.aillive.xyz/mail/
API 前缀：https://www.aillive.xyz/mail/api

日志目录：%LOCALAPPDATA%\AilliveMail\logs
```

线上入口应通过编译时配置写入正式版本，不能允许普通页面或用户输入任意服务器地址。若未来需要私有部署版，可增加经过校验的服务器配置功能。

## 17. 日志与隐私

客户端日志允许记录：

- 应用版本
- Windows 和 WebView2 版本
- 启动、更新、窗口和网络错误类型
- API 状态码和内部请求 ID

客户端日志禁止记录：

- 登录密码
- Cookie、Access Token、Refresh Token
- 邮箱密码和 Microsoft Refresh Token
- 完整邮件正文
- 附件内容
- 用户导入的账号原文

日志应支持大小轮转，并提供用户主动导出诊断日志的入口。

## 18. 安全基线

- 只允许 HTTPS。
- 主窗口仅允许 `www.aillive.xyz` 白名单路径。
- 第一阶段禁止配置 `dangerousRemoteDomainIpcAccess`。
- Tauri capabilities 使用最小权限。
- 禁止远程页面调用 Shell、文件系统和任意网络插件。
- Release 版本关闭开发者工具和远程调试端口。
- 外部链接交给系统浏览器。
- 更新包和安装程序均进行签名。
- 服务端继续执行用户身份校验和 owner scope 数据隔离。
- 客户端显示的数据不代表授权，所有敏感操作必须由服务端再次校验。

## 20. 验收标准

第一版 EXE 至少满足：

- Windows 10/11 x64 能正常安装和卸载。
- 双击图标只打开桌面窗口，不打开系统浏览器。
- 用户可以登录 `https://www.aillive.xyz/mail/` 的现有账号。
- 登录后可以读取服务器中属于该用户的邮箱数据。
- 未登录时显示与参考图结构一致的独立 Mail 登录页。
- 登录页背景飞机不会遮挡表单，并在登录成功后全部消失。
- 登录成功时品牌主飞机沿平滑路径飞入主界面左侧栏 Logo 位置。
- 飞机转场期间不出现登录页和主界面同时可操作的状态。
- 登录失败时保留登录页，不播放成功动画，也不清空用户已输入的邮箱。
- 关闭并重新打开客户端后登录状态仍然有效。
- 不允许站外页面替换主窗口。
- OAuth 和邮件中的外部链接使用系统浏览器打开。
- 同一时间只运行一个客户端实例。
- 客户端中不存在数据库文件和服务端密钥。
- 安装包可以在干净 Windows 环境完成安装并启动。

## 21. 风险与处理

### WebView2 未安装

Windows 10/11 通常已安装 Evergreen WebView2 Runtime。安装器仍需检测运行时，并在缺失时提示或引导安装。

### 远程站点不可用

第一阶段依赖线上页面。客户端应显示自定义连接失败页面，提供重新加载和网络诊断入口。

### Cookie 或登录失效

客户端必须正确使用固定 WebView2 用户数据目录，不在每次启动时创建临时目录。服务端会话被撤销后应正常回到登录页。

### 远程页面获得本机权限

第一阶段不向远程页面暴露 Tauri IPC。需要原生功能时优先由 Rust 自主执行，或迁移到第二阶段的本地可信前端。

### Windows SmartScreen

测试版本可以先内部使用；公开发布前应购买并配置代码签名证书，并持续使用同一发布者身份积累信誉。

## 22. 最终建议

采用以下路线：

1. 先实现 Tauri 远程安全壳，在较短时间内产出可安装、可登录、可正常收发邮件的 Windows EXE。
2. 不把 Go 服务和 SQLite 打包进客户端，继续使用 `https://www.aillive.xyz/mail/` 的线上服务和数据库。
3. 第一阶段不向远程页面开放 Tauri IPC，保证客户端本机权限边界。
4. 完成基础用户验证后，再迁移为本地 React 前端、Rust API Client 和桌面 Token 登录。
5. 正式公开发布前补齐自动更新签名、Windows 代码签名、日志隐私和干净系统安装测试。

这条路线能在开发速度、安装包体积、现有代码复用和长期桌面能力之间取得较好的平衡。
